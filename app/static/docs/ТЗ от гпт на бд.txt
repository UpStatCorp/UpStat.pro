Отлично. Этого уже достаточно, чтобы выдать сильное ТЗ первой полноценной версии.
Я сразу зафиксирую главный вывод: для UpStat нельзя строить систему как просто “таблицу из 150 столбцов”. Это будет тупик. У тебя список параметров частично плавающий, у части клиентов будут свои кастомные поля, часть источников данных будет неполной, а поверх всего этого должен работать ИИ-ассистент, который понимает голосовые и текстовые запросы и умеет сам считать агрегаты, сравнения, тренды и рекомендации. Значит, архитектура должна быть hybrid schema — гибридной схемой, а не жёсткой. Это прямо следует и из твоего описания задачи, и из внутренней логики UpStat, где система должна извлекать примерно 120–150 параметров разговора, считать вероятность сделки и потери, а также формировать рекомендации и обучение.
Ниже даю уже не размышления, а ТЗ.

ТЕХНИЧЕСКОЕ ЗАДАНИЕ
Проект: UpStat
Модуль: Structured Call Intelligence + AI Query Assistant
Версия: V1 Architecture Spec
________________


1. Цель модуля
Разработать модуль платформы UpStat, который:
1. принимает звонок, расшифровку, переписку или иной разговорный артефакт;
2. извлекает из него digital call avatar — цифровой аватар звонка, то есть набор структурированных параметров, сигналов, индексов и доказательств;
3. сохраняет результат в базу данных так, чтобы:
   * можно было быстро строить аналитику;
   * можно было хранить версии переанализа;
   * можно было добавлять новые параметры без миграций всей базы;
   * можно было поддерживать кастомные параметры разных клиентов;
4. даёт второму ИИ-слою возможность по голосовому или текстовому запросу:
   * искать нужные данные;
   * делать агрегаты;
   * сравнивать менеджеров;
   * строить тренды;
   * выявлять слабые сигналы;
   * генерировать рекомендации;
   * запускать уже существующий внутренний процесс тренировки;
   * формировать таблицы и данные для графиков;
   * отдавать ответ в текстовом виде и, при необходимости, в голосовом.
________________


2. Продуктовая логика, которую система обязана соблюдать
Система не должна ощущаться как “контроль менеджеров”. Она должна ощущаться как AI coach — ИИ-тренер и sales navigator — навигатор продаж. Внутри UpStat рекомендации не должны показываться как длинный список ошибок: менеджеру нужно показывать максимум 1–3 приоритетных действия, а владельцу — не детали звонков, а бизнес-метрики и тренды. Это прямо соответствует ранее сформулированной архитектуре продукта: показывать максимум три рекомендации, строить систему вокруг точки роста, а не вокруг обвиняющих ошибок, и разделять интерфейсы менеджера, РОПа и владельца.
Поэтому модуль должен строиться вокруг четырёх сущностей:
* signals — сигналы
* skills — навыки
* patterns — повторяющиеся паттерны
* actions — действия / рекомендации / тренировки
А не вокруг голых расшифровок.
________________


3. Поддерживаемые источники входных данных
3.1 Поддерживаемые типы источников
Система должна поддерживать входные данные следующих типов:
1. audio_call — запись звонка
2. transcript — готовая расшифровка
3. chat_screenshot_text — текст, извлечённый из скриншота переписки
4. chat_export — выгрузка переписки
5. manual_text — вручную вставленный текст разговора
6. crm_metadata — данные CRM, если есть
7. mixed_source — комбинация нескольких источников
3.2 Принцип отказоустойчивости к неполным данным
Система обязана корректно работать, если часть данных отсутствует.
Примеры:
* нет CRM;
* нет аудио, есть только transcript;
* нет transcript, есть только extracted text — извлечённый текст;
* нет данных по сделке;
* нет данных по менеджеру;
* нет результата звонка.
Во всех таких случаях:
* анализ не должен падать;
* отсутствующие поля должны храниться как null;
* параметры, которые нельзя надёжно вычислить, должны помечаться как:
   * unknown — неизвестно,
   * или low_confidence — невысокая уверенность.
________________


4. Ключевое архитектурное решение по данным
4.1 Что хранить по каждому параметру
Твой инстинкт “хранить всё сразу” в целом правильный, но в лоб это делать не надо. Если хранить вообще всё подряд без уровней важности, база быстро превратится в мусорный склад. Правильная модель — three-layer storage — трёхслойное хранение:
Слой 1. Core value — основное значение
Это то, что нужно для аналитики:
* число,
* флаг,
* enum — значение из списка,
* краткий текст,
* нормализованный score — балл.
Слой 2. Confidence + explanation — уверенность + краткое объяснение
Это нужно для доверия к модели:
* confidence_percent
* reasoning_summary — краткое объяснение
* data_source_type — источник данных
Слой 3. Evidence — доказательство
Это нужно для споров, аудита и обучения:
* цитата,
* таймкод,
* segment_id,
* при необходимости список фрагментов.
4.2 Что хранить НЕ надо по каждому параметру всегда
Не нужно по каждому параметру всегда хранить:
* длинное reasoning — длинную цепочку рассуждений;
* большие куски transcript;
* дублирование одних и тех же фрагментов 20 раз;
* полный raw prompt output — сырой ответ модели.
Это дорого и бесполезно.
4.3 Итоговое решение
По каждому параметру хранить надо:
* value_raw — сырое значение
* value_normalized — нормализованное значение
* value_type
* confidence_percent
* status (ok, unknown, low_confidence, needs_review)
* reasoning_summary — краткое объяснение в 1–3 предложениях
* evidence_refs — ссылки на доказательства
* source_segments — список сегментов, на которых основан вывод
Это лучший баланс между:
* точностью;
* проверяемостью;
* стоимостью хранения;
* скоростью аналитики.
________________


5. Ключевое архитектурное решение по базе данных
5.1 Рекомендуемый стек хранения
Для первой серьёзной версии рекомендую такой стек:
Основной вариант — рекомендованный
* PostgreSQL — Постгрес как основная база
* JSONB внутри PostgreSQL для гибких и кастомных параметров
* Object Storage — объектное хранилище для аудио, оригинальных транскриптов, JSON-артефактов анализа
* Redis для кэша быстрых запросов
* опционально позже:
   * ClickHouse — Кликхаус для сверхтяжёлой аналитики
   * Vector DB — векторная база для семантического поиска по разговорам
Почему не чисто SQL-таблица на 150 колонок
Потому что:
* список параметров будет меняться;
* часть параметров кастомные по клиентам;
* будут разные типы входа;
* будут версии моделей;
* будет переанализ;
* будут новые сигналы.
Почему не чистый NoSQL — NoSQL, нереляционная база
Потому что:
* тебе нужны фильтры, агрегаты, RBAC — разграничение ролей, мультитенантность и надёжные выборки;
* для бизнес-аналитики PostgreSQL сильнее и безопаснее.
5.2 Вердикт
Для UpStat V1 правильный баланс:
PostgreSQL + JSONB + Object Storage + Redis
Это даст:
* простоту запуска;
* гибкость;
* достаточную аналитическую мощность;
* возможность международного масштаба;
* лёгкий апгрейд позже.

7. SQL-логика хранения: главное решение
Архитектура должна быть dictionary-driven — управляемой словарём параметров.
То есть:
* новый параметр не требует добавлять новый столбец в таблицу conversations;
* новый параметр добавляется в parameter_definitions;
* значения пишутся в parameter_values;
* кастомные параметры клиента получают свой tenant_id.
Это самая важная инженерная развилка. Она делает систему живой, а не мёртвой.

8. Категоризация параметров
Параметры надо разбить минимум на 6 верхних блоков, потому что у вас уже есть продуктовая логика сигналов разговора:
1. control_of_conversation — контроль разговора
2. needs_discovery — выявление потребностей
3. deal_economics — экономика сделки
4. decision_process — процесс принятия решения
5. client_reaction — реакция клиента
6. closing_and_next_step — завершение и следующий шаг
Это полностью соответствует вашему внутреннему ядру сигналов: соотношение речи, количество вопросов, выявление проблемы, бюджет, критерии выбора, следующий шаг, вопросы клиента, снижение сопротивления, фиксация договорённости и т.д.
Дополнительно я бы ввёл ещё 4 системных блока:
7. metadata_quality — качество исходных данных
8. confidence_and_review — уверенность и необходимость проверки
9. training_relevance — пригодность для тренировки
10. prediction_indices — индексы вероятности сделки / риска потери

9. Индексы и производные метрики
Система должна хранить не только базовые параметры, но и derived metrics — производные метрики.
Минимально обязательны:
* deal_probability — вероятность сделки
* loss_probability — вероятность потери
* engagement_index — индекс вовлечённости клиента
* discovery_quality_index — индекс качества выявления потребностей
* objection_handling_index — индекс работы с возражениями
* control_balance_index — индекс баланса разговора
* next_step_strength_index — индекс силы следующего шага
* recommendation_pressure_index — индекс срочности коррекции
* skill_progress_delta — изменение навыка к прошлым звонкам
Это логически следует из вашей архитектуры: система должна не только извлекать параметры, но и формировать вероятность сделки, вероятность потери, ключевые сигналы и рекомендации.
________________


10. Правила хранения evidence — доказательств без взрыва ресурсов
Ты правильно уловил риск: evidence может раздуть хранение в 2–3 раза. Поэтому решение должно быть экономным.
10.1 Нельзя
* хранить полный transcript для каждого параметра;
* хранить одинаковую цитату 15 раз;
* хранить полные длинные reasoning-ответы модели.
10.2 Надо
Хранить transcript один раз по сегментам, а для параметров хранить только ссылки:
* segment_id
* start_ms
* end_ms
* short_excerpt до 200–300 символов
10.3 Правило
Для каждого параметра хранить:
* максимум 1–3 evidence refs — ссылок на доказательства;
* максимум 1 short excerpt;
* без дублей.
10.4 Итог
Такой подход почти не взрывает объём, потому что:
* основной текст уже лежит в conversation_segments;
* параметры только ссылаются на него;
* доказательства не дублируются.

11. Пайплайн анализа разговора
11.1 Этап 1. Ingestion — приём данных
Вход:
* аудио
* transcript
* chat
* CRM metadata
Результат:
* запись в conversations
11.2 Этап 2. Segmentation — сегментация
Если transcript уже есть:
* разбить на сегменты;
* привязать спикера;
* нормализовать таймкоды.
Если transcript нет, но есть аудио:
* ASR — автоматическое распознавание речи;
* diarization — разделение спикеров;
* сегментация.
11.3 Этап 3. Feature extraction — извлечение признаков
Извлечь низкоуровневые признаки:
* talk ratio — соотношение речи
* longest monologue — самый длинный монолог
* questions count — количество вопросов
* open questions count — количество открытых вопросов
* interruptions — перебивания
* pauses — паузы
* speech tempo — темп речи
* client response length trend — динамика длины ответов клиента
* keyword clusters — кластеры ключевых фраз
11.4 Этап 4. Parameter inference — вывод параметров
Для каждого параметра из parameter_definitions:
* вычислить значение;
* вычислить confidence;
* сформировать reasoning_summary;
* подобрать evidence refs;
* записать в parameter_values.
11.5 Этап 5. Signal inference — вывод сигналов
На основе параметров определить:
* позитивные сигналы
* негативные сигналы
* повторяющиеся паттерны
Например:
* не обсуждался бюджет;
* клиент стал говорить меньше;
* следующий шаг не зафиксирован;
* менеджер говорил более 70% времени.
Это полностью ложится на вашу спецификацию сигналов.
11.6 Этап 6. Skill scoring — оценка навыков
Преобразовать параметры и сигналы в skill scores — оценки навыков:
* выявление потребностей
* управление разговором
* экономика сделки
* работа с возражениями
* закрытие сделки
11.7 Этап 7. Recommendation engine — движок рекомендаций
Сгенерировать рекомендации не списком, а как действия:
* максимум 3;
* с приоритизацией;
* с привязкой к тренировке;
* с объяснением “почему сейчас”.
Это соответствует правилу трёх рекомендаций и действийному формату рекомендаций в вашей оргполитике.
11.8 Этап 8. Persistence — сохранение
Записать:
* analysis_version
* parameter_values
* signal_scores
* skill_scores
* recommendations
* outcomes
________________


12. Логика рекомендаций
Рекомендации должны ранжироваться так:
Приоритет 1
Ошибки/сигналы, напрямую влияющие на сделку:
* отсутствует попытка закрытия;
* не обсуждён следующий шаг;
* не выявлена проблема;
* клиент начал сопротивляться после цены.
Приоритет 2
Повторяющиеся паттерны:
* 5 звонков подряд не обсуждается бюджет;
* 4 звонка подряд мало открытых вопросов.
Приоритет 3
Редкие или косметические отклонения.
В ответ менеджеру:
* не писать “вы плохо сработали”;
* писать “следующий уровень навыка — …”.
Это жёстко соответствует вашей логике “точка роста, а не обвинение”.
________________


13. AI Query Assistant — ИИ-слой поверх базы
Теперь главное: как подключить второго ИИ, который будет отвечать на вопросы.
13.1 Архитектурная идея
Нельзя пускать LLM — большую языковую модель напрямую в базу без промежуточного слоя.
Нужен такой контур:
User Query → Intent Parser → Query Planner → Safe Data Access Layer → Calculation Layer → Recommendation Layer → Response Composer → UI Renderer
Перевод:
пользовательский запрос → парсер намерения → планировщик запроса → безопасный слой доступа к данным → слой расчётов → слой рекомендаций → компоновщик ответа → отрисовщик интерфейса.
13.2 Почему нельзя “просто дать ИИ доступ к SQL”
Потому что он:
* будет ошибаться;
* может строить неэффективные запросы;
* может нарушить границы ролей;
* может перепутать текущий tenant — клиента;
* может делать дорогие выборки.
13.3 Правильная модель
LLM не пишет SQL напрямую в production.
Она генерирует structured query plan — структурированный план запроса в JSON.
Потом backend:
* валидирует план;
* проверяет роль;
* применяет tenant filters — фильтры клиента;
* исполняет безопасные шаблоны SQL;
* считает агрегаты;
* только после этого отдаёт результат назад LLM для формулировки ответа.
________________


14. Поддерживаемые типы запросов AI-ассистента
Обязательные классы:
14.1 Aggregate queries — агрегатные запросы
Примеры:
* “За последние 5 звонков сколько в среднем процентов времени менеджер задавал вопросы?”
* “Какой средний индекс вовлечённости клиента за март?”
14.2 Comparison queries — сравнительные запросы
Примеры:
* “Сравни Ивана и Петра по выявлению потребностей за последние 30 дней.”
* “У кого лучше следующий шаг фиксируется?”
14.3 Trend queries — запросы по динамике
Примеры:
* “Покажи, как менялся индекс работы с возражениями за 3 месяца.”
* “Стало ли лучше по бюджету?”
14.4 Filter queries — запросы с фильтрацией
Примеры:
* “Найди звонки, где клиент был заинтересован, но следующий шаг не зафиксирован.”
* “Покажи звонки с низкой уверенностью анализа.”
14.5 Diagnostic queries — диагностические запросы
Примеры:
* “Какие три главные системные ошибки в отделе продаж?”
* “Что чаще всего мешает закрытию?”
14.6 Training queries — запросы на тренировку
Примеры:
* “Создай тренировку по самым частым ошибкам менеджера.”
* “Собери тренировку по работе с возражением ‘дорого’.”
14.7 Exploratory open queries — открытые исследовательские запросы
Примеры:
* “Что интересного ты видишь по нашим данным?”
* “Из-за чего чаще всего проседает разговор после обсуждения цены?”
Вот этот класс самый опасный. Для него нужна логика ограничений и уточнений.
________________


15. Режим уточнения
15.1 Принцип
Ассистент должен сам догадываться в 95% случаев.
15.2 Когда уточнять
Только если без уточнения:
* получится 10+ равновероятных трактовок;
* ответ будет почти наверняка бесполезен;
* или нарушится доступ по ролям.
15.3 Правило по умолчанию
Если пользователь не уточнил, то по умолчанию:
* период: последние 30 дней;
* выборка звонков: последние 20;
* менеджеры: доступный пользователю scope — периметр доступа;
* tenant: текущий клиент;
* метрика: наиболее вероятная трактовка из словаря параметров.
15.4 Формат уточнения
Одна короткая реплика:
“Я могу посчитать это по последним 5, 20 или всем звонкам за 30 дней. Чтобы не показать вам случайную выборку, уточните диапазон.”

18. RBAC — права доступа
owner — владелец
Видит:
* агрегаты по компании;
* тренды;
* сравнение команд;
* рекомендации высокого уровня;
* графики;
* бизнес-метрики.
По умолчанию не видит полные детали каждого звонка, если специально не открыл.
Это соответствует вашей логике: владелец должен видеть не детали звонков, а бизнес-метрики и влияние навыков на конверсию.
sales_head — РОП
Видит:
* команду;
* менеджеров;
* сравнительные графики;
* тренды;
* звонки своей команды;
* рекомендации;
* паттерны;
* тренировочные назначения.
analyst_upstat — аналитик UpStat
Видит:
* всё внутри клиента в пределах согласованного доступа;
* low confidence cases — случаи с невысокой уверенностью;
* версии анализа;
* служебные поля;
* параметры модели.
manager_limited — менеджер
Видит:
* только свои звонки;
* только свои навыки;
* только свои тренировки;
* без сравнения с другими, если это не разрешено отдельно.
________________


19. Версии переанализа
Хранить все версии надо обязательно.
Правило:
* каждая новая модель или новая логика создаёт новую analysis_version;
* старая версия не удаляется;
* одна версия помечается is_current = true;
* ответы ассистента используют текущую версию по умолчанию;
* аналитик UpStat может сравнить версии.
Это критично для:
* развития алгоритма;
* доказуемости;
* отладки;
* демонстрации улучшения модели;
* обратной совместимости.
________________


20. Работа с невысокой уверенностью
Если модель не уверена, параметр всё равно записывается.
Политика:
* confidence >= 80 → нормально
* 60–79 → невысокая уверенность
* < 60 → требует проверки
В ответах ИИ:
* если итог построен на значительной доле таких параметров, ассистент обязан указать:
“Часть вывода основана на данных с невысокой уверенностью модели.”
Не писать “низкая” без необходимости; формулировка “невысокая уверенность” действительно лучше для UX.
________________


21. Графики и визуализация
21.1 Принцип
ИИ не должен сам рисовать графики напрямую.
Он должен отдавать chart payload — полезную нагрузку для графика.
21.2 Рекомендованный вариант
Для веб-приложения рекомендую:
   * Highcharts если нужен богатый enterprise-уровень,
или
   * Apache ECharts если нужна сильная бесплатная альтернатива.
Если нужен мой жёсткий выбор:
      * Highcharts быстрее даст красивый SaaS-результат,
      * ECharts дешевле в лицензировании и гибче на длинной дистанции.
21.3 Что должен уметь backend
Формировать payload вида:
{
  "chart_type": "line",
  "title": "Динамика открытых вопросов",
  "x_axis": ["2026-01", "2026-02", "2026-03"],
  "series": [
    {
      "name": "Иван",
      "data": [18, 24, 31]
    },
    {
      "name": "Петр",
      "data": [14, 17, 16]
    }
  ],
  "meta": {
    "metric_code": "open_questions_count",
    "aggregation": "avg",
    "time_range": "monthly"
  }
}


21.4 Обязательные типы графиков
      * line — линия
      * bar — столбцы
      * stacked bar — накопленные столбцы
      * area — область
      * heatmap — тепловая карта
      * radar — радар навыков
      * funnel-like progression — воронкообразная динамика
      * scorecard / KPI cards — карточки KPI

22. Вызов внутренней тренировки
AI-ассистент не должен сам генерировать всю тренировку с нуля, если у вас уже есть внутренний engine — движок.
Правильная логика:
      1. ассистент диагностирует слабый навык;
      2. выбирает training_flow_id;
      3. вызывает internal API — внутреннее API;
      4. получает ссылку / идентификатор тренировки;
      5. возвращает пользователю:
      * краткое объяснение,
      * почему именно эта тренировка,
      * кнопку запуска.

26. Мой итоговый выбор по спорным местам
Даю уже мои жёсткие решения, чтобы разработчик не плавал:
         * база: PostgreSQL + JSONB
         * хранение файлов: object storage
         * кэш: Redis
         * модель параметров: dictionary-driven
         * хранение значений: parameter_values, не 150 fixed columns
         * evidence: через segment refs, не через дублирование текста
         * версии анализа: хранить все
         * режим уверенности: всегда записывать, но маркировать
         * AI-ассистент: через JSON query plan, не прямой SQL
         * графики: payload → Highcharts/ECharts
         * рекомендации: не больше 3 одновременно
         * владельцу: агрегаты и тренды, не поток сырых звонков
         * менеджеру: следующий шаг, а не список ошибок
Это всё согласуется и с твоим видением, и с уже сформулированной продуктовой логикой UpStat: система должна анализировать разговор, находить точки роста, давать тренировку и готовить к следующему разговору; владельцу показывать бизнес-метрики, а менеджеру — следующий шаг развития.

2. Полезные materialized views — материализованные представления
Чтобы AI-ассистент не долбил каждый раз по сырой таблице parameter_values, советую сразу сделать пару витрин.
2.1 current_analysis_per_conversation
CREATE MATERIALIZED VIEW current_analysis_per_conversation AS
SELECT
  c.conversation_id,
  c.tenant_id,
  av.analysis_version_id,
  c.manager_user_id,
  c.started_at,
  c.duration_seconds
FROM conversations c
JOIN analysis_versions av
  ON av.analysis_version_id = c.latest_analysis_version_id;


2.2 metric_flat_view
Это денормализованный слой для быстрых выборок.
CREATE MATERIALIZED VIEW metric_flat_view AS
SELECT
  c.tenant_id,
  c.conversation_id,
  c.manager_user_id,
  c.started_at,
  av.analysis_version_id,
  pd.code AS parameter_code,
  pd.category,
  pd.subcategory,
  pv.value_number,
  pv.value_text,
  pv.value_bool,
  pv.value_enum,
  pv.value_normalized,
  pv.status,
  pv.confidence_percent
FROM parameter_values pv
JOIN parameter_definitions pd ON pd.parameter_id = pv.parameter_id
JOIN analysis_versions av ON av.analysis_version_id = pv.analysis_version_id
JOIN conversations c ON c.conversation_id = av.conversation_id
WHERE av.is_current = TRUE;

3. Начальный словарь параметров
Тебе не надо ждать, пока будет весь список 150, чтобы запустить архитектуру.
Нужен parameter_dictionary_v1, который уже включает ядро.
Минимальные обязательные коды:

         * manager_talk_time_percent
         * client_talk_time_percent
         * longest_manager_monologue_seconds
         * manager_interruptions_count
         * client_interruptions_count
         * average_manager_replica_length_chars
         * average_client_replica_length_chars

         * manager_questions_count
         * manager_open_questions_count
         * manager_clarifying_questions_count
         * client_questions_count
         * question_time_percent
3.3 Needs discovery — выявление потребностей
         * pain_identified_flag
         * pain_depth_score
         * current_process_discussed_flag
         * desired_future_state_discussed_flag
         * decision_criteria_discussed_flag
         * budget_discussed_flag

         * next_step_proposed_flag
         * next_step_fixed_flag
         * specific_agreement_flag
         * implementation_discussed_flag
         * integration_discussed_flag
         * timeline_discussed_flag

         * client_engagement_index
         * client_response_length_growth_flag
         * client_interest_questions_flag

         * deal_probability

Это прямо бьётся с вашими блоками: клиент задаёт вопросы, обсуждает внедрение, интеграции, сроки, соглашается на следующий шаг, разговор заканчивается договорённостью; негативные сигналы включают отсутствие следующего шага и завершение разговора без договорённости.

6. SQL-шаблоны для AI Query Executor
LLM не должна писать SQL.
Она должна выбирать один из шаблонов.
________________


6.1 Aggregate metric template
SELECT
  AVG(mfv.value_number) AS metric_value,
  COUNT(*) AS rows_used
FROM metric_flat_view mfv
WHERE mfv.tenant_id = $1
  AND mfv.parameter_code = $2
  AND mfv.started_at IS NOT NULL
ORDER BY mfv.started_at DESC
LIMIT $3;


Для корректности в production лучше сначала выбрать нужные звонки CTE-блоком.
WITH last_calls AS (
  SELECT DISTINCT conversation_id, started_at
  FROM metric_flat_view
  WHERE tenant_id = $1
  ORDER BY started_at DESC
  LIMIT $2
)
SELECT
  AVG(mfv.value_number) AS metric_value,
  COUNT(*) AS rows_used
FROM metric_flat_view mfv
JOIN last_calls lc ON lc.conversation_id = mfv.conversation_id
WHERE mfv.tenant_id = $1
  AND mfv.parameter_code = $3;


________________


6.2 Compare managers template
SELECT
  u.display_name AS manager_name,
  AVG(mfv.value_number) AS metric_avg,
  COUNT(*) AS calls_count
FROM metric_flat_view mfv
JOIN users u ON u.user_id = mfv.manager_user_id
WHERE mfv.tenant_id = $1
  AND mfv.parameter_code = $2
  AND mfv.started_at >= $3
  AND mfv.started_at < $4
  AND u.display_name = ANY($5)
GROUP BY u.display_name
ORDER BY metric_avg DESC;


________________


6.3 Trend template
SELECT
  DATE_TRUNC('month', mfv.started_at) AS period,
  AVG(
    CASE
      WHEN mfv.value_bool IS TRUE THEN 1
      WHEN mfv.value_bool IS FALSE THEN 0
      ELSE NULL
    END
  ) AS share_true
FROM metric_flat_view mfv
WHERE mfv.tenant_id = $1
  AND mfv.parameter_code = $2
  AND mfv.started_at >= $3
  AND mfv.started_at < $4
GROUP BY 1
ORDER BY 1;


________________


6.4 Top negative patterns template
SELECT
  mfv.parameter_code,
  AVG(
    CASE
      WHEN mfv.value_bool IS FALSE THEN 1
      ELSE 0
    END
  ) AS failure_rate,
  COUNT(*) AS observations
FROM metric_flat_view mfv
WHERE mfv.tenant_id = $1
  AND mfv.parameter_code = ANY($2)
  AND mfv.started_at >= $3
  AND mfv.started_at < $4
GROUP BY mfv.parameter_code
HAVING COUNT(*) >= 5
ORDER BY failure_rate DESC, observations DESC
LIMIT 3;

9. Clarification policy — политика уточнений
Чтобы AI не бесил пользователя, у него должны быть жёсткие правила.
9.1 Не уточнять, если можно разумно предположить
Примеры:
         * “сравни последние звонки” → взять последние 20;
         * “покажи динамику” → взять 30 дней, если не указан период;
         * “по менеджерам” → взять доступную пользователю команду.
9.2 Уточнять только если:
         * выборка может radically alter outcome — радикально изменить результат;
         * непонятно, про какого менеджера речь;
         * запрос выходит за права доступа;
         * возможно 10+ трактовок.
9.3 Формат уточнения
Не спрашивать тупо:
“Уточните период.”
Писать так:
“Я могу посчитать это по последним 5, 20 или всем звонкам за 30 дней. Чтобы не показать вам случайную картину, уточните диапазон.”

13. Prompt architecture — архитектура промптов
Нужно не один промпт, а 4 разных класса промптов.
________________


13.1 Parameter extraction prompt
Задача:
         * по сегментам разговора вычислить значения параметров;
         * вернуть строго JSON.
13.2 Signal inference prompt
Задача:
         * по параметрам и сегментам вывести positive/negative signals.
В оргполитике прямо указано, что AI должен анализировать разговор одновременно по пяти измерениям: структура разговора, поведение менеджера, реакция клиента, динамика разговора и завершение разговора. Именно под это лучше и строить prompt decomposition — декомпозицию промптов.
13.3 Recommendation prompt
Задача:
         * не перечислить ошибки;
         * выбрать 1–3 лучших действия;
         * объяснить, почему именно они.
13.4 Query planning prompt
Задача:
         * из пользовательского запроса сделать JSON plan;
         * не отвечать пользователю напрямую;
         * не писать SQL.

ЧАСТЬ 3. ГОТОВЫЙ SEED, МИГРАЦИИ И СТРУКТУРА BACKEND
1. Мой жёсткий выбор по стеку
Для первой сильной production-версии я рекомендую:
FastAPI + PostgreSQL + SQLAlchemy 2.0 + Alembic + Redis + Pydantic + Celery/RQ + Object Storage
Почему я выбираю именно это, а не NestJS.
FastAPI здесь лучше по 5 причинам:
         1. У вас продукт AI-heavy — с упором на анализ, пайплайны, модели, JSON-контракты, фоновые джобы.
         2. Python естественнее для AI orchestration — оркестрации ИИ, обработки транскриптов, feature extraction и integration layer — слоя интеграции с LLM.
         3. Быстрее писать query planner, validators и analytics services.
         4. Легче связывать backend с ML/LLM логикой без лишних прокладок.
         5. Проще масштабировать модульно: анализ, запросы, рекомендации, тренировки.
NestJS я бы выбирал, если бы:
         * у вас был тяжёлый front/backend TypeScript-only контур,
         * команда уже очень сильна именно в Node,
         * а AI-часть жила бы как отдельный Python-микросервис.
Но для UpStat на текущем этапе FastAPI — более прямой и быстрый путь.

5. Seed file philosophy — философия seed-файла параметров
Вот здесь очень важный момент.
Нельзя делать seed просто как список из 150 названий параметров.
Он должен включать по каждому параметру:
         * code
         * title
         * description
         * category
         * subcategory
         * value_type
         * unit
         * default_aggregation_type
         * evidence_required
         * query_synonyms
         * normalization_rule
         * is_custom
Иначе AI-query слой потом не поймёт:
         * как эту метрику агрегировать,
         * как её искать по естественному языку,
         * как строить графики,
         * как понимать “похожую” формулировку запроса.

7. Как расширить seed до всех 150 параметров
Чтобы не превратить словарь в хаос, его надо расширять по секциям. Я рекомендую такой порядок.
Секция 1: control_of_conversation
Секция 2: questions
Секция 3: needs_discovery
Секция 4: deal_economics
Секция 5: decision_process
Секция 6: client_reaction
Секция 7: closing_and_next_step
Секция 8: presentation_quality
Секция 9: risk_signals
Секция 10: prediction_indices
Секция 11: confidence_and_review
Секция 12: training_relevance
Это не абстракция. У вас в оргполитике уже есть логика положительных и отрицательных сигналов: клиент обсуждает внедрение, интеграции, сроки, коллег и следующий шаг — это сильные сигналы сделки; не обсуждается следующий шаг, разговор заканчивается без договорённости, клиент просит “пришлите информацию” или уходит в неопределённость — это сигналы риска потери.

9. Service contracts — контракты сервисов
Чтобы проект не расползся, каждому сервису нужен чёткий интерфейс.
9.1 Parameter Engine
class ParameterEngine:
    async def compute_parameters(
        self,
        conversation_id: str,
        analysis_version_id: str,
        parameter_codes: list[str] | None = None
    ) -> list[dict]:
        ...


Возвращает список объектов такого вида:
{
  "parameter_code": "budget_discussed_flag",
  "value_bool": false,
  "value_normalized": 0.0,
  "status": "ok",
  "confidence_percent": 87.0,
  "reasoning_summary": "В разговоре не найдено прямого обсуждения бюджета или диапазона допустимых инвестиций.",
  "evidence_refs": []
}


9.2 Signal Engine
class SignalEngine:
    async def infer_signals(
        self,
        analysis_version_id: str
    ) -> list[dict]:
        ...


9.3 Recommendation Engine
class RecommendationEngine:
    async def generate_top_recommendations(
        self,
        analysis_version_id: str,
        max_items: int = 3
    ) -> list[dict]:
        ...


Это важно, потому что в вашей модели рекомендаций система должна делать не длинный список претензий, а выделять самые слабые сигналы и давать тренировку именно по ним.

from pydantic import BaseModel, Field
from typing import Literal, Optional, Any


class TimeRange(BaseModel):
    type: Literal["last_n_calls", "date_range", "relative_period", "quarter_to_date"]
    value: Optional[int] = None
    unit: Optional[Literal["day", "week", "month"]] = None
    from_date: Optional[str] = Field(default=None, alias="from")
    to_date: Optional[str] = Field(default=None, alias="to")


class SubjectFilter(BaseModel):
    field: str
    op: Literal["eq", "in", "gte", "lte"]
    value: Any


class Subject(BaseModel):
    entity: Literal["team", "manager", "managers", "calls", "department"]
    filters: list[SubjectFilter] = []


class Metric(BaseModel):
    parameter_code: str
    aggregation: Literal["avg", "sum", "count", "share_true", "latest", "distribution"]


class ChartConfig(BaseModel):
    needed: bool = False
    type: Optional[Literal["line", "bar", "area", "heatmap", "radar", "scorecard"]] = None


class AIQueryPlan(BaseModel):
    intent: Literal[
        "aggregate_metric",
        "compare_entities",
        "trend_metric",
        "diagnostic_summary",
        "launch_training",
        "filter_calls"
    ]
    tenant_scope: Literal["current"] = "current"
    role_scope: str
    time_range: TimeRange
    subject: Subject
    metric: Optional[Metric] = None
    metric_bundle: Optional[list[str]] = None
    comparison: Optional[str] = None
    group_by: Optional[str] = None
    chart: ChartConfig
    clarification_needed: bool = False
    assumptions: list[str] = []

12. Analysis orchestration — как должен идти запуск анализа
Последовательность должна быть такой:
         1. создаётся analysis_version
         2. подтягиваются conversation_segments
         3. считается feature extraction
         4. запускается parameter extraction
         5. сохраняются parameter_values
         6. выводятся signal_scores
         7. выводятся skill_scores
         8. считается deal_probability и loss_probability
         9. генерируются recommendations
         10. обновляется latest_analysis_version_id
         11. обновляются materialized views
Эта цепочка у вас полностью бьётся со стратегической логикой документа: система должна извлекать 120–150 параметров, формировать сигналы, вероятность сделки, вероятность потери и рекомендации.

13. Feature extractor — что считать до LLM
Очень важно не грузить LLM тем, что можно посчитать детерминированно.
До LLM нужно считать:
         * длительность речи менеджера
         * длительность речи клиента
         * ratio talk time — соотношение времени речи
         * среднюю длину реплик
         * longest monologue — самый длинный монолог
         * количество перебиваний
         * количество вопросов
         * количество открытых вопросов по rule-based + heuristic detection
         * паузы
         * изменение длины ответов клиента
         * появление вопросов со стороны клиента
         * позицию упоминания цены
         * позицию упоминания следующего шага
Это прямо следует из вашего документа: для детекции сигналов используются такие признаки, как ratio talk time, longest monologue, длина реплик, тональность, темп, паузы, перебивания.
То есть правильный инженерный подход:
         * всё, что можно извлечь алгоритмически, считаем кодом;
         * LLM используем только там, где нужна семантика и контекст.

14. Parameter extraction prompt design — как должен выглядеть prompt
LLM нельзя отправлять “проанализируй звонок как-нибудь”.
Нужно отправлять:
         * сегменты разговора,
         * список только нужных параметров,
         * строгий JSON schema output,
         * инструкцию не рассуждать свободным текстом,
         * инструкцию возвращать unknown, если уверенность низкая.
Пример логики промпта:
Ты анализируешь разговор менеджера и клиента.
Твоя задача — вернуть строго JSON-массив параметров.
Для каждого параметра верни:
- parameter_code
- value_number / value_bool / value_text / value_json
- value_normalized
- status
- confidence_percent
- reasoning_summary
- evidence_refs


Правила:
1. Не придумывай данные.
2. Если данных недостаточно, ставь status="unknown".
3. Если вывод вероятностный, но допустимый, ставь status="low_confidence".
4. Для evidence_refs используй только реально присутствующие segment_id.
5. Не возвращай ничего, кроме JSON.

15. Recommendation engine logic — точная логика рекомендаций
Вот здесь уже надо делать умно, а не банально.
Рекомендация должна строиться не по одному плохому звонку, а по формуле:
Severity — серьёзность
×
Repeatability — повторяемость
×
Business impact — влияние на сделку
×
Trainability — пригодность к тренировке
Псевдоформула:
recommendation_score =
(signal_weight * failure_rate * recency_weight * business_impact_weight * trainability_weight)


Где:
         * signal_weight — вес сигнала в модели,
         * failure_rate — как часто это повторяется,
         * recency_weight — насколько это свежая проблема,
         * business_impact_weight — насколько это бьёт по сделке,
         * trainability_weight — можно ли это реально исправить тренировкой.
Поэтому рекомендация “говорил слишком много” может оказаться ниже, чем “не фиксировал следующий шаг”, потому что второе ближе к прямой потере сделки. Это соответствует вашей внутренней логике негативных сигналов: отсутствие следующего шага и отсутствие договорённости — сигналы высокого риска потери.
________________


16. Training mapper — связь рекомендации и тренировки
Нужна таблица соответствий между слабым сигналом и обучающим модулем.
Например:
TRAINING_SIGNAL_MAP = {
    "next_step_fixed_flag": "training_close_next_step_v1",
    "budget_discussed_flag": "training_budget_conversation_v1",
    "pain_depth_score": "training_deep_discovery_v1",
    "manager_open_questions_count": "training_open_questions_v1",
    "decision_criteria_discussed_flag": "training_decision_criteria_v1"
}


То есть AI не должен каждый раз с нуля сочинять, чему учить.
Он должен:
         * найти слабый сигнал,
         * найти training flow,
         * запустить его.

17. Materialized view refresh strategy — стратегия обновления витрин
Это важный технический нюанс.
Не нужно обновлять материализованные представления после каждой записи одного параметра.
Нужно обновлять их:
         * после полного завершения анализа звонка;

         * по админ-команде;

Иначе backend сам себя задушит постоянными refresh.
